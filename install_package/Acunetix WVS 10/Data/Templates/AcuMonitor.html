AcuMonitor service

 

Some web vulnerabilities can only be detected or verified using an intermediate server. Acunetix Web Vulnerability Scanner version 9 introduces the detection of such vulnerabilities by using the AcuMonitor service. Acunetix WVS v9 users are required to register to the AcuMonitor service in order to detect and verify such vulnerabilities.

This article explains why the detection of certain vulnerabilities require the use of an intermediate server, how the AcuMonitor Service is used to detect these vulnerabilities, and which vulnerabilities are detected using this technique. The Email Header Injection vulnerability will be used to explain this further.

There are many web applications that implement a feedback submission web form. The web form accepts the user's email address, and a message. The web form creates an email, sets the FROM email address to the email address provided by the user, and the message is inserted in the email's body. The email's recipient is hard coded, so that all emails created from the web form are sent to the same email address within the organisation.

Email Header Injection is a common vulnerability for these type of web forms, whereby an attacker finds a way to inject additional email headers into the email through the web form. This is most often used by spammers, who inject additional recipients into the message header, thus making use of the organisation's web form to send spam emails.

When Acunetix WVS v9 scans such a web form, it instructs the web application to send an email to a unique email address hosted and monitored by the AcuMonitor Service. Each Email Header Injection request sent by Acunetix WVS during the scan is uniquely identified and stored in a database on the machine where Acunetix WVS is installed.

Web applications that are vulnerable to Email Header Injection will send an email to AcuMonitor. AcuMonitor will use the details in the message to identify the registered owner of the Acunetix WVS installation. AcuMonitor will automatically send a notification email to the registered owner with information retrieved from the email. The email notification will include the unique ID of the request that triggered the Email Header Injection. This ID can be used to identify the vulnerable web page and web form.

As you might appreciate, it would not be possible to automatically detect an Email Header Injection vulnerability without the use of the AcuMonitor Service acting as the intermediate server. There are other vulnerabilities, which will be explained further on, which require AcuMonitor.

The following diagram illustrates the steps needed to detect these type of vulnerabilities using Acunetix WVS and the AcuMonitor Service.

 

 

1.       The user initiates a scan. The user has already registered with the AcuMonitor Service from Acunetix WVS > Application Settings > AcuMonitor.

2.       Acunetix WVS starts scanning the website. As part of the scan, it will perform certain requests instructing the web application to make request to the AcuMonitor Service.

3.       Each request is stored in a database on the machine running Acunetix WVS, and given a unique ID.

4.       When a vulnerable web page or web form is found, the web application will make a request to the AcuMonitor Service. Depending on the vulnerability this can occur either during the Acunetix WVS scan, or after some time.

5.       AcuMonitor will process the request, identify the Registered User, and send a notification email to the user.

6.       The user will use the unique ID or the attachment in the notification mail to get more information on the request that triggered the vulnerability. This will allow the user to identify the vulnerable web page / web form.

 

Vulnerabilities identified with the help of AcuMonitor

·         Blind XSS

o   Various XSS script payloads are injected in the web application. The vulnerable web application will store the unsanitized script in a backend store (e.g. database or log file). When the unsanitized script is loaded, generally from a different web application, the script makes a web request to VVS which will send a notification email to the user. It generally takes a while for this to happen.

·         Email Header Injection

o   An email is sent from the vulnerable web application to AcuMonitor.

·         Server Side Request Forgery(SSRF)

o   An AcuMonitor URL is injected into the web application. The vulnerable web application will make a request to this URL. Acunetix WVS verifies with AcuMonitor if such request was made.

·         XML External Entity Injection

o   An XML containing an XXE entity pointing to the AcuMonitor domain is injected into the application. The vulnerable application will make a request to the VVS domain. Acunetix WVS verifies with AcuMonitor if such request was made.

·         Host Header based attacks

o   A malicious Host header pointing to the AcuMonitor domain is injected into the web application. If a request is later made to the AcuMonitor domain a notification email will be sent to the user.

AcuMonitor FAQ

Q. Why do I need to register to the AcuMonitor Service?
A. When a vulnerability is detected by AcuMonitor, AcuMonitor will send a notification email to the user registered with AcuMonitor with the details of the vulnerability. Registration links the scans done by an installation of Acunetix WVS to the user performing the scans.

Q. How do I register to the AcuMonitor Service?
A. On first launch of Acunetix WVS v9, you will be asked to register to AcuMonitor. Alternatively you can register from Acunetix WVS > Configuration > Application Settings > AcuMonitor > Register

Q. What is the 'Saved scans folder' used for?
A. The 'Saved scans folder' is used to store the web requests that are made to the scanned web applications which might trigger vulnerabilities that make use of VVS. This information is used to identify the vulnerable web page / web form. This information is never sent to the AcuMonitor. You can configure the location of the Saved scan folder from Acunetix WVS > Configuration > Application Settings > AcuMonitor.

Q. I have received an email notification about a vulnerability detected by AcuMonitor. What should I do?
A. The email notification sent to you by AcuMonitor contains information about the type of vulnerability that has been detected. Either double download and open the Report file, or open the attachment in order to load the web request which triggered the vulnerability. Alternatively, you can copy the Request  ID from the email, and lookup the web request manually from Acunetix WVS > Configuration > Application Settings > AcuMonitor > Lookup Request.

Q. Should I be concerned that information about the vulnerabilities on my website is hosted on AcuMonitor?
A. There is no need to be worried because of the following reasons:

·         The AcuMonitor Service is hosted in a very secure environment

·         The web request which was originally used to trigger the vulnerability is not stored at AcuMonitor. These are stored on the same machine running Acunetix WVS.

·         Any requests performed from your web application to AcuMonitor will only be stored for a limited amount of time (maximum 7 days).
